## 1. Project Vision

TL-AI is an AI-assisted platform dedicated **exclusively** to the game **Throne & Liberty**.

It provides a specification-driven, model-driven ecosystem in which the entire game domain
(entities, rules, mechanics) is captured as human-readable YAML specifications, compiled into
runtime artifacts, and consumed by four applications: **Core**, **Simulator**, **Studio**, and
**Lab**. Reasoning is provided by a locally hosted LLM (Qwen3 32B) orchestrated with LangGraph,
while all deterministic calculations are performed by a dedicated Mathematical Engine that is
strictly separated from the LLM.

## 2. Goals

- **Model the Throne & Liberty domain** completely and precisely as versioned YAML specifications.
- **Compile specifications** into validated runtime artifacts via the TL Compiler.
- **Simulate game mechanics** deterministically with an event-driven simulator.
- **Assist users with AI** through a local LLM (Qwen3 32B) orchestrated by LangGraph.
- **Maintain a local Knowledge Base** (PostgreSQL + pgvector) for retrieval-augmented reasoning.
- **Guarantee correctness** via JSON Schema validation and a Staging + Validation workflow.
- **Generate documentation** automatically from specifications.
- **Keep everything local-first**: local model, local knowledge base, local data stores.

## 3. Scope

In scope:

- **Throne & Liberty only** — all domain modeling targets this single game.
- **Four applications**: Core, Simulator, Studio, Lab.
- **Specification pipeline**: YAML specifications → JSON Schema validation → TL Compiler →
  runtime artifacts.
- **Mathematical Engine** for deterministic game-mechanic computation.
- **Event-Driven Simulator** for game-mechanic simulation.
- **Local Knowledge Base** with vector retrieval (PostgreSQL + pgvector).
- **Data acquisition tooling** based on Playwright.
- **Staging + Validation workflow** for all ingested and authored content.
- **Documentation generation** from specifications.

## 4. Non Goals

- **No other games** — the project will never support titles other than Throne & Liberty.
- **No cloud-hosted LLMs** — the model is local (Qwen3 32B); no external inference APIs.
- **No runtime YAML parsing** — the runtime never reads YAML (see AD-20).
- **No hand-written runtime domain data** — runtime artifacts are generated by the compiler only.
- **No hidden behavior** — no implicit conventions, no "magic" (see AD-23).
- **No game automation/botting** — TL-AI is an analysis, simulation, and assistance platform.

## 5. Architecture Overview

TL-AI follows **Specification Driven Architecture** and **Full Model Driven Architecture**:
the YAML specification corpus is the model of the domain; everything else is derived from it.

### 5.1 Specification Pipeline

```
YAML Specifications (one entity per file, immutable, stable IDs, PascalCase)
        │
        ▼
JSON Schema Validation
        │
        ▼
TL Compiler
        │
        ▼
Runtime Artifacts  ──►  consumed by Core / Simulator / Studio / Lab
```

- Specifications live in `specifications/` and are validated against schemas in `schemas/`.
- The **TL Compiler** is the only component that reads YAML; it generates runtime artifacts.
- The **runtime never reads YAML** — applications consume compiled artifacts exclusively.

### 5.2 Applications

| Application | Package | Responsibility |
|---|---|---|
| **Core** | `src/tl_core` | Central domain runtime and AI orchestration (LangGraph + Qwen3 32B) |
| **Simulator** | `src/tl_simulator` | Event-driven simulation of game mechanics |
| **Studio** | `src/tl_studio` | Authoring and management of specifications |
| **Lab** | `src/tl_lab` | Analysis and experimentation |

Supporting packages:

| Package | Responsibility |
|---|---|
| `src/tl_compiler` | TL Compiler: validates specifications and generates runtime artifacts |
| `src/tl_database` | Persistence layer: PostgreSQL, pgvector, SQLite |
| `src/tl_api` | API layer exposing platform functionality |
| `src/tl_common` | Shared utilities and cross-cutting concerns |

### 5.3 AI Layer

- **LangGraph** orchestrates agent workflows.
- **Qwen3 32B** runs locally and provides all LLM reasoning.
- The **Mathematical Engine is separated from the LLM**: the LLM never performs game-mechanic
  arithmetic; it delegates to the deterministic engine.
- The **Knowledge Base is local**: PostgreSQL with pgvector provides vector retrieval.

### 5.4 Data Flow and Validation

- All ingested and authored content passes through a **Staging + Validation workflow** before
  being promoted.
- **Playwright** is used for web data acquisition.
- **PostgreSQL** (with **pgvector**) is the primary datastore; **SQLite** serves local/embedded
  storage needs.

## 7. Development Principles

- **Human Readable First** — specifications and artifacts prioritize human readability.
- **Zero Magic** — no hidden conventions or implicit behavior; everything is explicit.
- **Immutable Specifications** — published specifications are never mutated in place;
  changes produce new versions.
- **One entity per YAML file** — each specification file defines exactly one entity.
- **Stable IDs** — every entity has a permanent, never-recycled identifier.
- **PascalCase naming** — domain entities and specification names use PascalCase.
- **Specification Driven** — code follows specifications, never the reverse.
- **Documentation generated from specifications** — domain docs are derived, not hand-written.
- **Runtime never reads YAML** — only the compiler consumes YAML.
- **Deterministic math** — game-mechanic computation belongs to the Mathematical Engine,
  never to the LLM.
